- 目的: キャリアを大切にする動機付けを強める。
- 変更: `CARRIER_INITIAL_LIVES` を `3 → 1`（`src/js/utils/Constants.js`）。
- 周辺整合:
  - リペアキットの「HP満タン時に +1 ライフ」ロジック（`main.js` 付近、`carrier.lives++`）は維持。

  - HOW TO PLAY 等の説明文（`ScreenRenderer.js`）に残機前提の記述があれば整合を確認。
    ライフ数を明示している箇所があれば更新。
  - HUD のライフ表示（`HUD.js`）はそのまま動作（値が1始まりになるだけ）。

## 2. 配点の見直し

すべて `src/js/utils/Constants.js` の定数変更を基本とし、スコア付与箇所は既存の
`addScore` / `game.score +=` を踏襲する。

### 敵撃破点

| 対象 | 現状 | 変更後 |
|------|------|--------|
| アタッカー standard | 100 | **300** |
| アタッカー heavy | 300 | **500** |
| アタッカー rival | 500 | **700** |
| アタッカー artillery | 800 | **900** |
| 巡航ミサイル (CRUISE_MISSILE_SCORE) | 100 | **150** |
| ドローン (ENEMY_DRONE_SCORE) | 150 | **250** |
| 地雷 (Landmine) 撃破 | 0/未設定 | **50** |

- 戦車(200)・砲台(200)・敵基地(10000)・フラグ(5000)・シールド破壊(50)は現状維持。

### ドローンの耐久力

- `ENEMY_DRONE_HP` を `5 → 8`（≒1.5倍、切り上げ）。撃破点250と合わせて手応えを上げる。

### 地雷・ホーミングミサイルのスコア付与

- Landmine: 破壊時に `addScore(50)` を付与（現状スコア付与が無ければ追加）。
- EnemyHomingMissile: プレイヤーに撃墜された時に `addScore(10)` を付与
  （現状は撃墜時スコアなし。artillery が撃つホーミングミサイルを撃ち落とす行為を評価）。
  - 自然消滅（lifetime切れ）や自爆・着弾では加点しない。プレイヤーの攻撃で破壊された場合のみ。

### アイテム取得

- リペアキット / ミサイルキット / オートエイムユニットの取得時に **各200点**。
  - 定数 `ITEM_PICKUP_SCORE = 200` を新設し、各アイテムの取得（pickup）処理で `addScore` する。

## 3. ゲームモード（ノーマル / ニュータイプ）

### 目的

- **ノーマル**（デフォルト）: 0.8倍速のウェイトで弾幕をよけやすく、じっくり遊べる。
- **ニュータイプ**: 現状どおりウェイトなし。実時間あたりの進行が速く、タイムボーナスを稼ぎやすい。

### モード選択UI

- タイトル画面で **←/→キー** によりモード切替。選択中モードを画面に表示する。
- デフォルトは **ノーマル**。
- 状態: `Game.mode`（`'normal' | 'newtype'`）、`Game.gameSpeed`（`normal=0.8`, `newtype=1.0`）。
- 既存タイトル操作（Tab=carrierLift、任意キー/クリックで開始）とキーが競合しないこと。

### 速度の実装方式（固定タイムステップ・アキュムレータ）

現状、物理はフレーム単位の固定ステップ（`x += vx`、deltaTime非依存）。
タイマー（`totalTime`/`missionTimer`）だけが実 `deltaTime` を積算している。

- `loop()` で `simAccumulator += deltaTime * gameSpeed` を積算。
- `STEP = 1000/60 (≈16.67ms)` ごとに物理更新（`_updatePlaying` のシミュレーション部分）を1ステップ実行。
- タイマー（`totalTime`/`missionTimer`）は **生の `deltaTime`** で進める（＝「時間の進み方は同じ」）。
- ニュータイプ(1.0)は現状の挙動をほぼ維持。ノーマル(0.8)は物理が20%スロー。
- 入力・単発処理（キーのエッジ検出、武器切替、発射トリガ、ドッキング判定など）は
  1フレームに1回だけ実行し、物理ステップ反復ループの外に置く（多重発火防止）。
- 副次効果: 高リフレッシュ環境（120/144Hz）で速く動く現状挙動が是正され、速度が
  リフレッシュレート非依存になる。

### カクつき対策（カメラ補間から着手）

- 素のアキュムレータは描画が「整数ステップ位置」に限られ、0.8倍では平均5フレームに1回
  “動かないフレーム”が入り微小なカクつきが出る。
- **本フェーズのスコープ**: カメラの描画位置のみ補間する。
  - `alpha = simAccumulator / STEP` を用い、カメラ描画位置 = `lerp(前ステップのカメラ位置, 現ステップのカメラ位置, alpha)`。
  - 体感的なカクつきの大半は背景スクロール由来のため、これで大きく改善する見込み。
- **将来拡張（本フェーズ未実装）**: 不足時に全エンティティ `prevX/prevY` を持たせた完全補間へ拡張。

### タイムボーナス再設定

- 現状（`_onFlagCaptured`）: `targetTimeBonus = max(0, baseBonus - floor(missionTimer/1000)*50)`、
  `baseBonus = floor(totalTiles/100)*100`。
- 方針: **敵の殲滅より速攻クリアを高く評価**する（ハイリスクハイリターン）。減衰は特に急峻化しない。
  - **基準額を引き上げる**（例: `baseBonus` を約1.5倍）。→ 速く抜ければ撃破稼ぎより大きく伸びる。
  - **減衰率はモード別**とし現状水準を維持: ニュータイプ `50 pt/秒`、ノーマル `40 pt/秒`
    （ノーマルは0.8に合わせ甘め）。急峻化はしない。
- 減衰率・基準倍率は定数で保持し調整可能に
  （例: `TIME_BONUS_BASE_MULT = 1.5`、`TIME_BONUS_DECAY = { normal: 40, newtype: 50 }`）。
- ニュータイプのタイムボーナス優位は意図的に残す（完全な等化はしない）。

## テスト方針

既存 Node テスト（`tests/`、`node --test`）に純関数中心のテストを追加。

- タイムボーナス計算: 基準倍率適用、モード別減衰（normal=40 / newtype=50）、下限0クランプ。
- モード切替の状態遷移: ←/→ で `mode`/`gameSpeed` が期待通り変わる。
- アキュムレータ/カメラ補間: ステップ反復回数の算出、`lerp` を純関数化して単体テスト。
- 配点は定数変更が主で、スコア加算経路（Landmine撃破・ホーミング撃墜・アイテム取得）の
  分岐を純関数/薄いヘルパに切り出せる範囲でテスト。
